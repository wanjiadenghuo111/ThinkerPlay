5W1H 看推理

把“模型为什么慢、为什么吃显存、为什么并发上不去”拆成六个问题，更容易定位硬件瓶颈。

What

推理是什么

加载训练好的权重，把输入 token 变成下一个 token 的概率分布，再循环生成答案。

Why

为什么依赖硬件

大部分时间在矩阵乘和注意力上，容量不足会 OOM，带宽不足会拖慢每个 token。

Who

谁在参与

CPU、系统内存、GPU/NPU、显存共同工作；四者任一短板都会限制延迟、吞吐或上下文。

When

什么时候消耗最多

Prefill 消耗计算和显存带宽，Decode 反复读权重和 KV cache，高并发时缓存增长最快。

Where

数据在哪里流动

请求和 token 在内存，张量进入显存，算子在 GPU/NPU，采样和返回再回到 CPU。

How

如何优化

选合适模型规模，量化权重，限制上下文，控制并发，使用批处理、paged attention 和 KV cache 策略。

四大硬件关系

推理性能不是单看某个部件峰值，而是看数据能否稳定穿过整条链路。

CPU：控制面

负责请求生命周期、分词、调度、采样、日志与网络 I/O。高并发或复杂后处理会让 CPU 先排队。

  • 影响排队时间和首 token 前准备
  • CPU-only 推理可行但通常吞吐低

内存：承接面

保存服务进程、请求队列、token、权重映射和 CPU offload 数据。内存不足会触发交换或直接失败。

  • 影响稳定性和并发承载
  • 离线加载大模型时很关键

GPU/NPU：计算面

执行矩阵乘、注意力、归一化等核心算子。GPU 更通用，NPU 常在 INT8/INT4 上能效高但算子支持更挑剔。

  • 影响 token/s 和 prefill 速度
  • 还受内存带宽和 kernel 实现约束

显存：容量面

承载模型权重、KV cache、激活和临时 workspace。显存通常决定可跑多大模型、多长上下文、多高并发。

  • 权重受量化影响最大
  • KV cache 随上下文和并发线性增长

资源依赖矩阵

同一项任务会同时消耗多个硬件维度，表中的“强/中/弱”表示最常见的压力来源。

推理要素 CPU 内存 GPU/NPU 显存
模型权重加载 中解析、映射、调度加载 强文件缓存、mmap、offload 弱加载后参与计算 强权重常驻，量化直接降低占用
Prefill 长 prompt 中分词和提交任务 中token 与请求缓冲 强大规模并行计算 强注意力、激活、KV 写入
Decode 逐 token 中采样、停止条件、流式输出 弱小量状态数据 强重复执行每层算子 强反复读取权重和 KV cache
上下文长度 弱分词时间随 token 增长 中请求与缓存元数据增长 中注意力计算随序列变重 强KV cache 近似线性增长
并发和批处理 强队列、调度、采样压力上升 强会话状态和缓冲成倍增长 强利用率上升，延迟也可能上升 强每个会话都占 KV cache
量化与算子支持 弱主要负责选择路径 中权重文件和加载峰值降低 强取决于 kernel 和硬件指令 强权重占用和带宽压力下降

估算器中的 KV cache 使用“层数 × 上下文 token × 并发 × K/V × KV 头数 × head dim × KV 精度”近似计算；推荐规格会额外预留稳态余量。实际结果还会因 GQA/MQA、paged attention、框架实现、batch 策略和具体硬件 kernel 而变化。

快速判断规则

遇到延迟、OOM 或吞吐不足时，可以先按下面的症状定位。

显存爆了

先降模型规模或权重量化，再缩短上下文和并发；KV cache 通常是长上下文的核心变量。

首 token 很慢

重点看 prompt 长度、prefill 吞吐、CPU 分词和排队；长文档场景可考虑缓存 prompt。

生成 token/s 低

重点看 GPU/NPU 利用率、显存带宽、量化 kernel、batch 策略和模型大小。

并发上不去

重点看 CPU 调度、系统内存、KV cache 总量和请求队列；吞吐优化常伴随单请求延迟上升。
